tekton-dag handles the full lifecycle today — build, test, intercept, merge, release. Here is what we are building next to make it production-grade on real infrastructure.

The first pillar is retry on transient failures. On cloud infrastructure — especially with spot instances — nodes disappear mid-build. A Kaniko push times out because the registry is momentarily unreachable. A DNS lookup fails during a burst of concurrent pipelines. These are not code bugs. They are infrastructure noise. The plan is to add task-level retries to build, deploy, and containerize tasks. Critically, test and validation tasks will not retry — if a test fails, that is a real signal. The system will distinguish between eviction exit codes and application exit codes, and structured annotations on each TaskRun will record the retry count and original failure reason for post-mortem analysis.

The second pillar is precise build image sizing. Today, build tasks use default resource requests. On a shared cluster, that means either wasted capacity — Maven requesting four gigabytes when it needs one — or OOMKill when a large Spring Boot monolith exceeds the default. We are adding per-tool resource profiles configurable through Helm values. Maven gets a different CPU and memory profile than npm. Kaniko gets its own profile separate from compilation. Stack YAML will allow per-app overrides for outliers. And a monitoring baseline script will capture peak usage per TaskRun to inform sizing decisions with real data.

The third pillar is multi-cluster push. Today, tekton-dag builds and deploys within a single cluster. Production environments need released images promoted to separate clusters — staging, production, disaster recovery. We are adding a registry list in configuration — each entry names a target registry with its credential secret. A new promotion pipeline will take a release version and a target environment, pull the image from the build registry, push it to the target, and optionally trigger the deployment. Between environments, an optional approval gate will require a manual sign-off — a slash approve comment on the PR or a button in the management GUI. Every promotion will be recorded in Tekton Results for full audit trail.

The fourth pillar is operational reliability. Explicit pipeline timeouts will prevent tasks from hanging indefinitely on a slow cluster. The finally block will still execute on timeout, ensuring intercept cleanup and PR status comments happen even when something goes wrong. Health-check gates after deployment will wait for pod readiness probes before running tests. And backup scripts for both the Tekton Results Postgres database and the Neo4j graph will ensure persistent data survives cluster incidents.

The fifth pillar is observability. Prometheus metrics will track build duration per tool, test pass rates, retry counts, and pipeline queue time. Sample alerting rules will fire when the failure rate exceeds a threshold or registry pushes start failing. Cost attribution labels on TaskRun pods — team, stack, and app — will feed cluster cost allocation tools like Kubecost, so teams know exactly what their pipelines cost.

Infrastructure-grade reliability. Cost-aware sizing. Multi-environment promotion. Full observability. That is milestone thirteen — production hardening for tekton-dag.
